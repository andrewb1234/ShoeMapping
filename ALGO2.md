Your current approach in `shoe_clustering.py` has a subtle flaw common to K-Means: **the hard boundary problem**. Right now, your code finds the cluster for the query shoe, and *only* looks for neighbors within that exact same cluster. If a shoe sits right on the edge of Cluster A, its true closest neighbor might be just across the border in Cluster B, but your algorithm is completely blind to it.

Here is how you can build a vastly superior matching algorithm using synthetic supervised learning, without doing any manual grading, and without the need for clunky browser automation.


**The Pipeline:**
1. Randomly sample pairs of shoes from your database.
2. Feed their JSON `lab_test_results` into an LLM prompt.
3. Ask the LLM to output a structured JSON score (e.g., 0 to 100).

**Example Prompt for the LLM:**
> "You are an expert running shoe fitter. I am going to give you the lab specs for Shoe A and Shoe B. 
> Shoe A: {shoe_a_specs}
> Shoe B: {shoe_b_specs}
> Rate how good of an alternative Shoe B is for a runner who loves Shoe A on a scale of 0 to 100. Consider drop, stack height, foam softness, and weight. Output ONLY a JSON object like {"similarity_score": 85}."

Run this script overnight for 10,000 random pairs, and you suddenly have a massive, high-quality supervised dataset without doing any manual work.

Once you have this LLM-generated dataset of shoe pairs and their "Match Scores" (0-100), you abandon K-Means entirely. You are no longer evaluating "centroid distance" or "label proximity." 

Instead, you train a model to predict the similarity score between any two shoes.

**The Architecture (Pairwise Regression):**
For every pair in your dataset, calculate the absolute difference between their features. 
* `delta_drop = abs(shoe_A_drop - shoe_B_drop)`
* `delta_weight = abs(shoe_A_weight - shoe_B_weight)`
* `delta_softness = abs(shoe_A_softness - shoe_B_softness)`

You then train a Gradient Boosting model (like **XGBoost** or **LightGBM**) or a simple Random Forest. 
* **Input Features:** The `delta_` values between the two shoes.
* **Target Variable:** The 0-100 score generated by the LLM.

If you move to this supervised approach, you stop evaluating based on geometric distances and start evaluating based on **Ranking Metrics**.

When a user selects "Adidas Adistar", your web app will:
1. Pair the Adistar with every other shoe in the database.
2. Calculate the feature differences (`deltas`).
3. Run those differences through your XGBoost model to get a predicted 0-100 score for every shoe.
4. Sort by the highest score and show the top 5 to the user.

**The Evaluation Metric:**
You evaluate this using **NDCG (Normalized Discounted Cumulative Gain)** or **MAE (Mean Absolute Error)**. You test a hold-out set of shoe pairs and check: *Did my XGBoost model give high scores to the same pairs that the LLM expert gave high scores to?*
